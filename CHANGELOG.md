## 2026-07-23
- Шаблонные уведомления клиентам: бухгалтер ВИДИТ предстоящие сообщения бота, правит/прикладывает документы; иначе бот шлёт по расписанию (мост чтения/записи к QA-платформе, миграция 0035).
  - **Что.** Приложение бухгалтера получило раздел «Уведомления» (`/notifications`, всем) и управленческий обзор «Рассылки (день)» (`/notifications-daily`, только менеджеры). Данные живут в QA-платформе (repo #1, `mqa_notification_*`, `mqa_chats.language`); читаем через `kk_*` views, пишем через `kk_*` SECURITY DEFINER RPC — точь-в-точь мост нарушений (0027).
    1. **Чтение со скоупом (0035):** клиенточувствительный контент (планы на 30 дней, вложения, лог отправленных) НЕ выставлен anon-view — иначе любой держатель anon-ключа читал бы уведомления всех клиентов. Читается через login-code SECURITY DEFINER RPC `kk_list_planned_notifications` / `kk_list_notification_attachments` / `kk_list_sent_notifications`, которые возвращают ТОЛЬКО компании звонящего (супервайзер — все; владение считается в БД через `kk_notification_scope`/`kk_owns_contract`). `kk_notification_templates` (общий каталог, не клиентские данные) остаётся view; `kk_chat_directory` расширен только `language` (карта бухгалтер→договор anon НЕ раскрывается).
    2. **RPC (0035):** `kk_edit_notification` (правка текста — АУДИТ в `mqa_notification_edits`, без «тихих» правок), `kk_approve_notification`, `kk_cancel_notification`, `kk_attach_notification` (файл по месяцу / отметка «сделано» + сопроводительный текст). Все с логин-код аутентификацией и серверной проверкой владения чатом (`kk_assert_chat_owner`: свой клиент ИЛИ супервайзер), гранты только anon/authenticated на EXECUTE.
    3. **UI:** `src/pages/Notifications.jsx` — по компаниям: карточка каждого предстоящего уведомления с явным предупреждением «это БУДЕТ отправлено» (`WILL_SEND_WARNING`), кнопки Редактировать/Подтвердить/Отменить, ручная секция вложения для MANUAL-типов, и read-only лог «отправленные» на клиента. `NotificationsDaily.jsx` — все уведомления, сгруппированные по дню отправки (`groupByDay`), с отменой (п.5). Пункты в `nav.js` + маршруты в `App.jsx`.
  - **Зачем.** ТЗ владельца/переворот логики: платформа планирует рассылки, бухгалтер их видит и при необходимости правит/прикладывает; шаблонные сообщения уходят только ботом, не вручную.
  - **Как.** Чистая логика — `src/lib/notifications.js` (статусы/лейблы/группировка по дням/`WILL_SEND_WARNING`), доступ — `src/lib/api.js` (`fetch*`/`edit`/`approve`/`cancel`/`attach*`, шлют `getStoredCode()`). Живая клиентская отправка выполняется в repo #1 и по умолчанию ВЫКЛючена (гейт `approved`+`NOTIFICATIONS_SEND_ENABLED`).
  - **Результат.** `npm test` — 285 зелёных (новые `notifications.test.js` — 9, `notificationApi.test.js` — 9; обновлён `nav.test.js`); `npm run build` — успешно. Новый файл: `supabase/migrations/0035_kk_notifications_bridge.sql` (гард на prerequisite repo #1 `20260723_mqa_notifications_v1.sql`).
